## 3. Issues found

### Issue 1 — `IndexError` in M5 analysis when saturation guard truncates the schedule (minor)

**Location:** `scripts/milestone5_crossing.py`, `stage_analyse`, line ~181.

**Problem:** The offset-corrected bootstrap is called with a truncated `sweeps[(sweeps <= upper)]` array but with full-length `hot`/`cold` estimator arrays. Inside `offset_corrected_difference_bootstrap`, `mask = times > cutoff` therefore has a different length than `traj_mean`, causing `IndexError`.

**Reproduction:** Run the M5 pipeline at `N = 64` with the primary pair; the cold leg hits the saturation guard (`L_S/N ≈ 0.405 > 0.40`), producing `upper = 6425 < 8000 = sweeps[-1]`.

**Impact:** Does **not** affect the committed `N = 128` result (`upper = 15000 = sweeps[-1]`). It does mean the pipeline is not robust to configurations where the saturation guard clips the schedule. Fix: slice `hot`/`cold` to the same indices used for `sweeps`, or pass the full sweeps and mask inside the bootstrap routine.

**Severity:** minor (robustness, not a headline conclusion).

## 4. Overall assessment

The headline conclusion — **no Mpemba-like inversion in the studied 2D conserved Kawasaki system at the operational point `M = 0`, `T_f = 0.6 T_c`, primary pair `T_i = 10` vs `2.4`, `N = 128`** — stands as scoped. The result is a clean, statistically controlled negative finding supported by two independent methods:

1. A pre-registered crossing search with bootstrap confidence intervals, BH-FDR control, offset correction, and a two-of-three estimator rule; and
2. A 4×4 exact-diagonalisation spectral probe showing the hotter preparation has *more* slow-mode overlap (anti-Mpemba), consistent with the larger-`N` no-inversion verdict.

The pre-registration, append-only provenance, validation suite, and open documentation meet a high reproducibility standard. The one actionable flaw I found is the latent schedule-truncation `IndexError`, which should be fixed but does not invalidate the published `N = 128` conclusion.

For publication readiness, I would require only:
- Fix the saturation-guard truncation bug; and
- Add a regression test or documented guard so the M5 pipeline fails gracefully (or runs correctly) when `upper < t_max`.

Everything else — model correctness, statistical protocol, scope discipline, and reproducibility — I judge to be sound.
